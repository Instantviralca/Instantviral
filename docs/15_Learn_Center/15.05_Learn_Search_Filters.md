# InstantViral v2

# Document 15.05 — Learn Search & Filters

Version: 1.0
Status: Production Specification

## Purpose

Create a fast, accessible, SEO-safe search and filtering system for the Learn Center.

The system should help users discover published articles by:

- Search query
- Category
- Tag
- Platform
- Reading time
- Publication date
- Updated date
- Featured status

The architecture must work with the shared Learn article registry, Categories & Tags System, Internal Linking Engine, Analytics System, and SEO Metadata Engine.

---

## Public Surfaces

Support search and filters on:

- /learn
- /learn/[category]
- Optional future /learn/search
- Optional future tag pages

Do not create indexable duplicate pages for every filter combination.

---

## Objectives

- Fast client-side discovery for the current article set
- Server-side filtering readiness for 100+ articles
- Accessible controls
- Mobile filter drawer
- Clear empty states
- Stable sorting
- Consent-aware analytics
- SEO-safe query parameter handling
- No draft or unpublished article leakage

---

## Searchable Fields

Search across:

- Article title
- Excerpt
- Category
- Tags
- Platform
- Author name
- Keywords
- Optional heading index
- Optional short-answer summary

Do not search private editorial notes or draft-only metadata.

## Search Input

Requirements:

- Accessible label
- Placeholder: Search articles
- Search icon
- Clear button
- Debounced input
- Keyboard accessible
- Escape key may clear when appropriate
- No page reload
- Search state reflected in results count
- Preserve search during filter changes

Recommended debounce:

200–350ms

Do not fire analytics on every keystroke.

## Result Display

Display:

- Result count
- Active filters
- Article cards
- Empty state

- Pagination or Load More readiness

---

## Search Result Ordering

When query exists:

1. Exact title match
2. Title keyword match
3. Tag or category match
4. Excerpt match
5. Other indexed text match

When query is empty:

Use the selected sort order.

Ensure deterministic results for equal scores.

---

## Search Normalization

Normalize:

- Case
- Leading and trailing spaces
- Repeated spaces
- Basic punctuation
- Safe Unicode handling

Optional future support:

- Synonyms
- Typo tolerance
- Stemming

Do not over-engineer fuzzy search for Version 1.

---

## Search Index Architecture

For Version 1:

- Build a lightweight index from published article metadata
- Do not ship full article bodies to the client unnecessarily
- Include only safe searchable fields

Prepare future provider abstraction for:

- Server-side database search
- Meilisearch
- Algolia
- Typesense
- Another provider

Do not integrate an external search provider yet.

---

## Query Parameters

Suggested parameters:

- q
- category
- tag
- platform
- readingTime
- sort
- featured
- page

Requirements:

- Parse safely
- Validate against active registries
- Ignore unsupported values
- Do not allow script injection
- Keep URLs shareable where useful
- Prevent uncontrolled parameter combinations

---

## SEO Rules

Default Learn and category pages remain indexable.

Filtered and search states should generally:

- Keep canonical on the clean base or category route
- Avoid being added to the sitemap
- Avoid generating indexable duplicate pages
- Use noindex when a dedicated search-results route is created
- Not generate separate metadata for every arbitrary query

Do not block normal users from using filters.

Canonical and robots behavior must be explicit and tested.

## Empty States

### No Search Results

Heading:

No articles found

Copy:

Try a different search term, remove a filter, or browse one of the Learn categories.

Actions:

- Clear Search

Do not generate placeholder articles.

## Accessibility

Ensure:

- Search has a visible or screen-reader label
- Results count uses an appropriate live region
- Mobile drawer traps focus
- Focus returns to the filter trigger
- Controls are keyboard operable
- Chips have descriptive remove labels
- Sort has a proper label
- Loading state is announced where needed
- No keyboard traps
- 44px minimum touch targets
- 200% zoom support

---

## Responsive Rules

- Search at top
- Sidebar or horizontal filters
- 3-column article cards where space permits

Tablet:

- Compact filter bar
- 2-column cards

Mobile:

- Full-width search
- Filter drawer
- Single-column cards
- Sticky Apply button only if it does not obscure content
- No horizontal overflow

---

## Performance

- Search article metadata, not full bodies
- Memoize filtered results where appropriate
- Debounce input
- Avoid expensive work on every render
- Keep client bundle small
- Prepare server-side filtering for large datasets
- Lazy-load images
- Avoid third-party search scripts

---

## Security

- Sanitize query parameters
- Validate categories and tags
- Reject unsafe values
- Do not expose drafts
- Do not expose private author fields
- Do not render arbitrary HTML from queries
- Protect future server search endpoints with rate limiting
