# Document 15.02 — Article Template System

Version: 1.0
Status: Production Specification

## Purpose

Create one reusable production article template for every Learn Center article.

The template should support:

- Long-form SEO content
- Clear reading flow
- Table of contents
- Author information
- Featured images
- Content blocks
- FAQs
- Related articles
- Related services
- Article schema
- Internal linking
- Responsive reading experience
- Future editorial expansion

Do not build articles as one-off page layouts.

---

# 1. Public Route

Use the Learn article route defined by the existing architecture.

Recommended:

`/learn/[slug]`

If the project already uses a different approved route structure, preserve it consistently.

---

# 2. Article Page Structure

Use this order:

1. Breadcrumb
2. Article Hero
3. Article Metadata
4. Featured Image
5. Table of Contents
6. Main Article Content
7. Key Takeaways
8. Related Service CTA
9. Article FAQ
10. Author Box
11. Related Articles
12. Final CTA
13. Footer

Optional blocks should render only when content exists.

---

# 3. Article Hero

Support:

- Category eyebrow
- H1
- Excerpt or introduction
- Author
- Published date
- Updated date
- Reading time
- Optional featured label

Rules:

- One H1 only
- Title must come from article data
- Dates must be real
- Reading time must be calculated or stored consistently
- Do not display an update date when no meaningful update occurred

---

# 4. Article Metadata

Display:

- Author name
- Published date
- Last updated date when applicable
- Estimated reading time
- Category
- Optional tags

Use semantic `<time>` elements.

Do not fabricate authors or dates.

---

# 5. Featured Image

Recommended dimensions:

- 1600 × 900 px
- 16:9 ratio
- WebP or AVIF

Requirements:

- Use `next/image`
- Meaningful alt text
- Width and height provided
- Priority only when it is the LCP image
- Caption and credit support
- Do not use copyrighted images without permission

Fallback:

Use an approved branded placeholder only during development.

---

# 6. Table of Contents

Generate the TOC from article headings.

Support:

- H2 items
- Optional H3 nesting
- Anchor links
- Active section highlighting
- Keyboard navigation
- Mobile collapse/expand
- Sticky desktop behavior where usable

Rules:

- Heading IDs must be unique
- Do not include hidden headings
- Do not create empty links
- Account for sticky-header scroll offset

---

# 7. Main Content Width

Recommended reading width:

- 700–780px for the primary article column

Desktop may use:

- Main content column
- Sticky TOC/sidebar

Avoid extremely wide paragraphs.

---

# 8. Supported Content Blocks

Create typed content blocks for:

- Paragraph
- Heading
- Bulleted list
- Numbered list
- Image
- Figure with caption
- Blockquote
- Callout
- Tip
- Warning
- Definition
- Comparison table
- Data table
- Step-by-step process
- Key takeaway box
- FAQ group
- Internal CTA
- Related service card
- Related article card
- Divider
- Embedded media placeholder
- Code block when relevant

Do not allow arbitrary unsanitized HTML.

---

# 9. Content Block Data Model

Each block should support:

- id
- type
- content
- headingLevel where relevant
- image metadata where relevant
- links
- styleVariant
- order
- schemaRelevant where applicable

Use discriminated TypeScript unions.

No `any`.

---

# 10. Paragraph and Heading Rules

Article writing should use:

- Short paragraphs
- Clear headings
- Descriptive subheadings
- Direct answers near the beginning
- Natural keyword use
- Practical examples
- No keyword stuffing
- No unsupported claims

Heading hierarchy:

- One H1
- H2 for main sections
- H3 for subsections
- Avoid skipping levels

---

# 11. Lists

Lists should be used only when they improve scanning.

Requirements:

- Complete and parallel phrasing
- No unnecessarily long lists
- Semantic `<ul>` and `<ol>`
- Clear lead-in text
- No list used only for visual indentation

---

# 12. Images and Figures

Each image block should support:

- src
- width
- height
- alt
- caption
- credit
- sourceUrl where permitted
- decorative flag
- loading priority

Rules:

- Alt text describes the image’s purpose
- Decorative images use empty alt text
- Captions should not repeat alt text word for word
- External image hotlinking should be avoided
- Verify usage rights

---

# 13. Tables

Support accessible tables.

Requirements:

- Visible caption where useful
- Semantic header cells
- Scope attributes
- Responsive overflow wrapper
- No table used only for layout
- Mobile readability

For large comparison tables, provide a simplified mobile presentation where needed.

---

# 14. Callouts

Support:

- Information
- Tip
- Warning
- Important
- Example

Each callout must include:

- Icon or label
- Accessible text
- Semantic structure
- Design-system token styling

Do not rely on colour alone.

---

# 15. Key Takeaways

Optional block near the end of the article.

Recommended:

- 3–6 concise takeaways
- No new claims not supported by the article
- Useful for readers who skim

Do not duplicate the introduction word for word.

---

# 16. Related Service CTA

Articles may link to relevant production services.

Rules:

- Use the Internal Linking Engine
- Maximum 1 prominent related-service CTA in the main article body
- Additional service links should be contextual and restrained
- Link only to approved active service pages
- Do not link every article to every service
- Do not use aggressive or misleading CTA copy

Examples:

- Explore Instagram Followers Packages
- View TikTok Views Packages
- Compare YouTube Subscriber Options

Do not use generic “Click Here.”

---

# 17. Article FAQ

Optional.

Use only when the article contains genuinely useful follow-up questions.

Rules:

- Store FAQ data in the shared FAQ system or article data
- Visible FAQ content and FAQPage schema must match
- Do not create FAQ schema for questions not rendered
- Avoid repeating the article’s main headings as fake FAQs

---

# 18. Author Box

Display:

- Author name
- Role
- Short bio
- Avatar when approved
- Author profile link when available

Do not invent credentials, education, job titles, or expertise.

Author information must come from the Author System.

---

# 19. Related Articles

Display 3–4 relevant published articles.

Priority:

1. Same topic cluster
2. Same platform
3. Complementary user intent
4. Recently updated relevant content

Do not include:

- Drafts
- Unpublished articles
- Current article
- Unrelated articles
- Placeholder content

Use the Related Articles Engine when implemented.

---

# 20. Reading Progress

Optional client component.

Requirements:

- Lightweight
- Does not block rendering
- Respects reduced motion
- Accurate progress calculation
- Hidden from screen readers if purely decorative
- Does not overlap mobile controls

---

# 21. Share Links

Optional.

If enabled, support:

- Copy link
- LinkedIn
- X/Twitter
- Facebook
- Email

Rules:

- Use the canonical URL
- Do not expose tracking parameters by default
- External links use safe attributes
- Do not load heavy social SDKs

---

# 22. Newsletter CTA

Optional and disabled by default.

Only enable when:

- A real newsletter system exists
- Consent wording is ready
- Privacy Policy is updated
- Email provider is configured
- Unsubscribe process works

Do not render a non-functional newsletter form.

---

# 23. Article SEO

Each article should support:

- Unique title
- Unique meta description
- Canonical URL
- Open Graph metadata
- Twitter/X card
- Article schema
- BreadcrumbList
- Organization
- Author entity
- Featured image
- Published and modified dates

Metadata must match visible content.

---

# 24. Article Schema

Support fields such as:

- headline
- description
- image
- datePublished
- dateModified
- author
- publisher
- mainEntityOfPage

Rules:

- Use genuine dates
- Use verified author data
- Use the canonical URL
- Use the visible featured image
- Do not output fake ratings
- Do not include FAQPage unless visible FAQs exist

---

# 25. Published and Modified Dates

Published date:

- Original public publication date

Modified date:

- Meaningful content revision date

Do not update `dateModified` for:

- Build deployments
- Formatting changes only
- Unrelated code changes
- Automatic daily rebuilds

---

# 26. Canonical Rules

Every article should have one clean self-referencing canonical URL.

Do not canonicalize different articles to:

- The Learn homepage
- A category page
- Another article
- The homepage

Avoid duplicate slug variants.

---

# 27. Internal Linking Rules

Each article should include relevant links to:

- 2–5 related articles when available
- 1–3 related service pages where contextually appropriate
- Relevant FAQ or policy pages where necessary
- Category page
- Learn index via breadcrumb

Avoid excessive exact-match anchor repetition.

---

# 28. External Sources

When factual claims require sourcing:

- Prefer official sources
- Prefer primary research
- Use descriptive anchor text
- Verify that links work
- Open in a new tab only when appropriate
- Use safe `rel` attributes
- Do not cite low-quality sources for important claims

Do not add sources only for decoration.

---

# 29. Content Safety and Accuracy

Articles must not:

- Promise guaranteed social media growth
- Promise rankings
- Promise monetization
- Promise sales or revenue
- Present purchased engagement as organic activity
- Invent statistics
- Misrepresent platform policies
- Copy competitor content
- Use fake quotes
- Use fake author credentials

---

# 30. Responsive Layout

Desktop:

- Main article and optional sidebar
- Sticky TOC when space permits
- Comfortable reading width

Tablet:

- Single primary column
- TOC above article or collapsible

Mobile:

- Single column
- Full-width images
- Responsive tables
- Large link and accordion targets
- No horizontal overflow
- No sticky element covering content

---

# 31. Accessibility

Verify:

- One H1
- Logical headings
- Semantic article element
- Accessible TOC
- Descriptive links
- Alt text
- Accessible tables
- Visible focus states
- Reduced-motion support
- Colour contrast
- Keyboard operation
- 200% zoom support
- No inaccessible embedded media

---

# 32. Performance

- Server-render main content
- Minimize client components
- Lazy-load below-fold images
- Optimize featured images
- Avoid heavy social embeds
- Avoid excessive JavaScript
- Generate TOC server-side where practical
- Keep reading-progress code lightweight
- Use caching for published articles

---

# 33. Empty and Error States

Support:

- Article not found
- Draft/unpublished article
- Missing author
- Missing image
- No related articles
- No related service
- Invalid content block

Drafts must not be publicly indexable.

Invalid production data should be reported rather than silently replaced with fabricated content.

---

# 34. Preview Mode

Prepare future editorial preview support.

Preview pages must:

- Require authorized access
- Be noindex
- Not appear in sitemap
- Not expose draft content publicly
- Use visible preview status for editors
