# 12.01 Admin Architecture

**Version:** 1.0  
**Module:** Admin  
**Status:** Planned

## 1. Purpose

Define a secure, maintainable admin application for operating InstantViral orders, customers, products, payments, notifications, and reporting.

## 2. Architecture Principles

- [x] Keep admin functionality separate from public customer routes.
- [x] Use server-enforced authorization for every protected action.
- [x] Organize modules by operational domain.
- [x] Share consistent navigation, data views, and action patterns.
- [x] Make role- and permission-based access straightforward to introduce and extend.
- [x] Preserve auditability for staff actions affecting orders, payments, and customers.

## 3. Route Structure

All admin routes are grouped beneath `/admin` and require authenticated staff access.

| Route | Module | Purpose |
| --- | --- | --- |
| `/admin` | Dashboard | Operational overview and prioritized work. |
| `/admin/orders` | Orders | Search, filter, review, and manage orders. |
| `/admin/orders/[orderId]` | Orders | Order details, timeline, fulfilment, and notification history. |
| `/admin/customers` | Customers | Customer lookup and order history. |
| `/admin/products` | Products | Product catalog and fulfilment configuration. |
| `/admin/payments` | Payments | Payment, refund, and reconciliation review. |
| `/admin/notifications` | Notifications | Delivery history, failures, and resend operations. |
| `/admin/reports` | Reports | KPI and operational reporting. |
| `/admin/settings` | Settings | Controlled operational and integration settings. |
| `/admin/team` | Team | Staff users, roles, and permissions when enabled. |

Routes may be released incrementally; absent modules should not be represented by insecure placeholders.

## 4. Module Boundaries

### 4.1 Dashboard
Aggregates read-optimized operational data and routes staff to actionable queues.

### 4.2 Orders
Owns order list and detail experiences, status history, fulfilment actions, internal notes, and safe links to related records.

### 4.3 Customers
Provides customer-scoped support context while minimizing display of personal data.

### 4.4 Payments
Provides payment events, refunds, transaction references, and reconciliation states. Payment credentials and sensitive instrument data are never exposed.

### 4.5 Notifications
Provides delivery status, provider callbacks, retry visibility, and authorized manual resend controls.

### 4.6 Reports
Provides filtered aggregate metrics; exports and time ranges must be permission-aware.

### 4.7 Settings and Team
Provides configuration and staff administration, restricted to appropriate elevated permissions.

## 5. Shared Components

- [ ] `AdminShell` — authenticated layout, navigation, and global page frame.
- [ ] `AdminSidebar` — permission-aware module navigation.
- [ ] `AdminHeader` — page title, breadcrumbs, global search, and staff menu.
- [ ] `DataTable` — pagination, sorting, filtering, loading, empty, and error states.
- [ ] `FilterBar` — reusable filters with URL-synchronized state.
- [ ] `StatusBadge` — standardized order, payment, and delivery statuses.
- [ ] `MetricCard` — dashboard KPI display with comparison and trend affordances.
- [ ] `Timeline` — immutable event presentation for order and payment histories.
- [ ] `ConfirmActionDialog` — explicit confirmation for consequential actions.
- [ ] `AuditPanel` — actor, action, target, timestamp, and reason display.
- [ ] `EmptyState`, `ErrorState`, and `LoadingState` — consistent operational feedback.

## 6. Layout

The admin layout contains a persistent desktop sidebar, compact mobile navigation, a top header, and a bounded content area. Navigation highlights the active module and hides unauthorized destinations. Page-level actions are placed consistently in the header or local action area.

Data-heavy pages use a predictable sequence: title and actions, filters, summary counts, table or detail content, then pagination or related activity. Layout must remain usable at common desktop and tablet widths and preserve keyboard navigation.

## 7. Permissions-Ready Design

Authorization must be enforced by server-side route guards and action handlers; client-side visibility is only a usability layer.

### 7.1 Initial permission vocabulary

- `dashboard.view`
- `orders.view`, `orders.update`, `orders.cancel`
- `customers.view`
- `payments.view`, `payments.refund`
- `notifications.view`, `notifications.resend`
- `reports.view`, `reports.export`
- `settings.view`, `settings.manage`
- `team.view`, `team.manage`

### 7.2 Role mapping

Initial implementations may map all authenticated administrators to a single administrator role. The authorization boundary must nevertheless evaluate named permissions, enabling future roles such as Support, Operations, Finance, Analyst, and Super Admin without route rewrites.

## 8. Audit and Safety Requirements

- [ ] Log every mutation with actor, action, target, timestamp, before/after summary, and reason when required.
- [ ] Require confirmation for cancellations, refunds, resends, and destructive settings changes.
- [ ] Avoid logging sensitive customer, payment, and secret values.
- [ ] Protect mutation endpoints against CSRF and replay where applicable.
- [ ] Provide clear errors without exposing internal implementation details.
- [ ] Use pagination and scoped queries for operational performance.

## 9. Acceptance Checklist

- [ ] All `/admin` routes require staff authentication.
- [ ] Every admin action has server-side permission enforcement.
- [ ] Navigation and actions reflect permissions without being the authorization source.
- [ ] Shared data and status components are used across modules.
- [ ] Consequential mutations are auditable and confirmed.
- [ ] Unauthorized and missing resources receive safe, consistent responses.
