# 12.04 Service Management

**Version:** 1.0  
**Module:** Admin  
**Status:** Planned

## 1. Purpose

Manage the public service catalog while keeping service identity, available packages, published content, and discoverability consistent with the service registry.

## 2. Route and Permissions

- Route: `/admin/services` with editor route `/admin/services/[serviceId]`.
- [ ] Require `settings.view` (or a future `services.view`) to read and `settings.manage` (or `services.manage`) to publish changes.
- [ ] Validate all saved content server-side and retain actor, before/after summary, and publication timestamp.

## 3. Service Table

The table lists registry-backed services; it does not let staff invent unsupported platform/category combinations.

| Column | Content |
| --- | --- |
| Service | Name, slug, platform, and category. |
| Availability | Active, hidden, draft, or unavailable state. |
| Pricing | Count of active real packages and missing-pricing warning. |
| Homepage | Featured state and display order. |
| SEO | Metadata completeness indicator. |
| Updated | Last editor and timestamp. |
| Actions | Edit, preview, and permitted publish controls. |

- [ ] Search by name, slug, platform, and category.
- [ ] Filter by platform, category, publication state, homepage feature, and pricing readiness.
- [ ] Paginate, sort, and provide explicit empty/error states.

## 4. Service Editor

The editor uses tabs with unsaved-change protection, validation summaries, draft/published state, and a clear save/publish flow.

### 4.1 Basics

Shows immutable service ID/slug where registry-owned, name, platform, category, short description, active state, order-field requirements, and fulfilment configuration references. Changes that affect existing orders require dedicated migration approval and may not be silently applied.

### 4.2 SEO tab

Manage title, meta description, canonical path, social title/description/image, and indexability. Display character guidance but validate semantic requirements rather than truncating content unexpectedly.

- [ ] Canonical and social URLs resolve to the public service page.
- [ ] Do not permit indexing of unpublished or unavailable services.

### 4.3 Pricing tab

Displays packages resolved from the real InstantViral catalog for the service platform/category. It shows package ID, name, quantity, USD price, popular flag, active/availability state, and sort order.

- [ ] Do not create fabricated tiers, quantities, or prices.
- [ ] Package selection/configuration may only reference catalog IDs matching this service.
- [ ] Warn when a service has no real matching packages; do not publish a misleading price section.

### 4.4 Homepage tab

Manage whether the service is eligible for homepage placement, feature state, display order, card title/description overrides, and optional image/icon subject to asset rules. The public homepage must show only active, publishable services.

### 4.5 Related tab

Select related services from active catalog entries. Prevent self-links, duplicates, unpublished targets, and unrelated categories when relevance rules require a match. Ordering is explicit and stable.

### 4.6 FAQ tab

Manage ordered question-and-answer entries for the public service page. Require a non-empty question and sanitized answer; allow draft entries. FAQ structured data must derive only from published, visible entries.

## 5. Preview

Preview renders the current unsaved draft in an isolated, non-indexable context using the same public page components and package resolver. It includes desktop/mobile widths and highlights unresolved references, missing pricing, SEO omissions, and unpublished related services.

- [ ] Preview never writes changes or changes public caches.
- [ ] Previewed packages use the same formatting and availability rules as production.
- [ ] Publish validates again server-side; preview success is not approval to bypass validation.

## 6. Acceptance Checklist

- [ ] The table exposes service readiness without inventing catalog data.
- [ ] Each editor tab saves validated, auditable content.
- [ ] Pricing only resolves packages matching the selected service.
- [ ] Homepage, related-service, and FAQ configuration excludes invalid targets.
- [ ] Preview faithfully renders the draft without exposing it to search engines.
