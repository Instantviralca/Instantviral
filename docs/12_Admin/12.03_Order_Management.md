# 12.03 Admin Order Management

**Version:** 1.0  
**Module:** Admin  
**Status:** Planned

## 1. Purpose

Provide authorized staff a fast, auditable workspace to locate orders, understand fulfilment state, communicate internal context, and perform controlled operational actions.

## 2. Route and Permissions

- Route: `/admin/orders`; an optional deep link may use `/admin/orders/[orderId]`.
- [ ] Enforce `orders.view` for all reads, `orders.update` for non-terminal status changes, and `orders.cancel` for cancellations.
- [ ] Enforce permissions in server handlers; hidden UI controls are not authorization.
- [ ] Record each mutation with actor, timestamp, previous and new value, and an operator reason where required.

## 3. Search and Filters

The search field accepts a full or partial order ID, customer email, payment reference, provider order reference, and submitted target URL or handle where available. Debounce client input, but execute the scoped query server-side.

Filters are URL-synchronized and include:

- [ ] Status: pending, paid, processing, completed, partial, failed, cancelled, refunded, and requires-action.
- [ ] Payment status and payment provider.
- [ ] Service, platform, package, and fulfilment provider.
- [ ] Created, paid, updated, and completion date ranges.
- [ ] Coupon code, customer, staff owner, manual-review flag, and exception state.
- [ ] Amount range and currency where multi-currency support is enabled.

A clear-all action restores the default queue. Saved views may be introduced later, but must preserve permission-scoped results.

## 4. Orders Table

The table is server-paginated, sortable, keyboard operable, and retains filters during navigation. Default sort is newest created order first; operational queues may explicitly choose oldest-first.

| Column | Content |
| --- | --- |
| Order | Short ID with copy action and full ID in accessible label. |
| Created | Timestamp and configured operational timezone. |
| Customer | Email or safe customer identifier. |
| Service / package | Service name, platform, selected package, and quantity. |
| Amount | Charged total, currency, discount, and payment state. |
| Fulfilment | Provider/reference and delivered versus ordered quantity when known. |
| Status | Standardized status badge with text label. |
| Updated | Last operational event timestamp. |
| Actions | Open details; consequential actions only after confirmation. |

- [ ] Provide loading, empty, error, and no-results states.
- [ ] Do not expose payment instrument data, secrets, or unredacted provider payloads.
- [ ] Make row selection separate from opening the details drawer.

## 5. Details Drawer

Opening a row presents a URL-addressable, focus-managed drawer without losing the list context. The drawer contains summary, customer-safe contact context, ordered service and package, configured order fields, payment summary, fulfilment details, discount/coupon details, and links to permitted related records.

### 5.1 Status controls

Status transitions follow the lifecycle rules, not arbitrary edits. Display allowed next transitions, validation errors, and confirmation for terminal or customer-impacting changes.

- [ ] Require a reason for cancellation, refund-related status changes, failures, partial completion, and manual fulfilment overrides.
- [ ] Prevent completion above ordered quantity and cancellation after an incompatible terminal state.
- [ ] Queue customer notifications according to the notification policy and show whether delivery succeeded.

### 5.2 Timeline

The immutable timeline combines order creation, payment events, fulfilment submissions and callbacks, status changes, notifications, refunds, coupons, and staff actions. Each event shows timestamp, actor or system source, safe summary, and correlation/reference IDs when permitted.

### 5.3 Internal notes

Notes are staff-only, attributed, timestamped, and append-only by default. Support rich text only if sanitized. Editing or deletion, if enabled, must preserve revision/audit history.

## 6. Bulk Actions Architecture

Bulk actions apply only to explicitly selected rows in the current permission scope. Selection is ID-based, displays the selected count, and never silently targets every result across pages.

| Layer | Responsibility |
| --- | --- |
| Table selection | Captures eligible order IDs and communicates ineligible rows. |
| Action policy | Declares action permissions, status preconditions, confirmation copy, and reason requirements. |
| Preview endpoint | Revalidates access and returns eligible, skipped, and blocked orders before execution. |
| Job executor | Performs idempotent, per-order mutations with concurrency limits and audit events. |
| Results panel | Reports success, skipped, and failed IDs with retry-safe remediation links. |

Initial bulk candidates: assign owner, add internal tag/note, retry eligible fulfilment, and export permitted data. Bulk cancellation, refund, or status changes require explicit policy approval, per-order validation, a typed confirmation, and asynchronous results.

## 7. Acceptance Checklist

- [ ] Staff can search and filter orders without escaping their authorization scope.
- [ ] Table values and status badges match the source order record.
- [ ] The details drawer preserves list state and exposes an auditable timeline and notes.
- [ ] Status changes obey lifecycle validation and notification rules.
- [ ] Bulk actions validate every order server-side and report partial outcomes.
- [ ] Sensitive payment and customer information remains minimized and protected.
