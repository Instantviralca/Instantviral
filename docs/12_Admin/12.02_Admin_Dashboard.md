# 12.02 Admin Dashboard

**Version:** 1.0  
**Module:** Admin  
**Status:** Planned

## 1. Purpose

Provide an actionable operational overview for authorized staff. The dashboard prioritizes work that needs attention over passive reporting.

## 2. Audience

- Operations staff monitoring order fulfilment
- Support staff resolving customer issues
- Finance staff reviewing payments and refunds
- Administrators monitoring service health and business performance

Displayed widgets and actions must respect the viewer's permissions.

## 3. Dashboard Layout

1. Header with date range, refresh state, and permission-aware quick actions.
2. KPI row for core business and operational metrics.
3. Attention queue for exceptions requiring staff action.
4. Trend and volume widgets for recent performance.
5. Recent operational activity and system signals.

The default time range is today, with clear timezone labeling. Supported ranges include today, 7 days, 30 days, and a custom range where permitted.

## 4. KPI Widgets

| Widget | Definition | Primary use |
| --- | --- | --- |
| Orders | Count of orders created in the selected range | Demand volume monitoring. |
| Revenue | Captured payment total in the selected range | Sales performance. |
| Average order value | Captured revenue divided by paid orders | Basket value trend. |
| Completion rate | Completed orders divided by eligible completed/cancelled orders | Fulfilment performance. |
| Processing orders | Current count of orders in progress | Operational workload. |
| Failed payments | Count and rate of payment failures | Checkout/payment issue detection. |
| Refunds | Count and value of refunds | Financial and service-quality monitoring. |
| Notification failures | Failed or suppressed transactional messages | Customer communication health. |

Each metric card shows the selected value, period comparison when meaningful, and a link to the relevant filtered module.

## 5. Operational Widgets

### 5.1 Attention queue

Prioritized cards or table rows for:

- [ ] Orders requiring customer action
- [ ] Fulfilment failures or delayed processing
- [ ] Payment failures requiring review
- [ ] Pending refunds
- [ ] Notification delivery failures
- [ ] Unread or unassigned support-related items when integrated

Each item includes its severity, age, owner when applicable, and a direct link to the relevant record.

### 5.2 Order volume and status trend

Shows order volume over time with status breakdowns. It supports operational planning and should not obscure exact values behind inaccessible visual-only controls.

### 5.3 Revenue and refund trend

Shows captured revenue and refund activity over the selected period. Finance-only details must remain permission-gated.

### 5.4 Fulfilment health

Shows current processing count, average processing duration, partial completions, completion rate, and failed/delayed orders.

### 5.5 Recent activity

Shows a chronological feed of safe, high-value events: new orders, status changes, refunds, manual staff actions, and notification failures. Every item links to its originating record when authorized.

### 5.6 System and integration health

Shows concise integration indicators for payments, fulfilment, and notifications. It reports degraded or unavailable states without exposing secrets, raw credentials, or unsafe provider payloads.

## 6. Quick Actions

Quick actions are visible only where the viewer holds the corresponding permission:

- [ ] Search orders
- [ ] View processing orders
- [ ] Review failed payments
- [ ] Review pending refunds
- [ ] View notification failures
- [ ] Create or manage products
- [ ] Open reports
- [ ] Manage settings

Actions must take the user to an appropriately filtered page rather than performing high-impact mutations directly from the dashboard.

## 7. Data and Refresh Behavior

- Dashboard metrics use a server-defined timezone and clearly label it.
- Summary values should load quickly from aggregate or read-optimized queries.
- The page shows the last successful refresh time.
- Manual refresh is available to authorized users and must avoid duplicate requests.
- Background refresh may be added later; it must not overwrite an active user filter or hide errors.
- Empty, unavailable, and partial-data states are explicitly represented.

## 8. Permissions and Privacy

- [ ] Enforce `dashboard.view` server-side for the route and data queries.
- [ ] Gate each widget's data and destination according to relevant module permissions.
- [ ] Do not expose customer emails, payment details, or secrets in aggregate widgets.
- [ ] Hide finance metrics from users without finance/report permissions where required.
- [ ] Log exports and high-sensitivity drill-down actions.

## 9. Accessibility and Usability

- [ ] Provide text equivalents for trends, chart values, and status color meanings.
- [ ] Maintain keyboard access to all widget actions and drill-down links.
- [ ] Use readable dates, localized currency formatting, and explicit empty states.
- [ ] Preserve filter state in the URL when appropriate.
- [ ] Make the dashboard responsive without removing critical attention items.

## 10. Acceptance Checklist

- [ ] Authorized users see KPIs, attention items, trends, and recent activity for their allowed scope.
- [ ] Dashboard values match the source modules for the same range and filters.
- [ ] Attention items link to actionable, permission-checked records.
- [ ] Quick actions never bypass confirmation or authorization requirements.
- [ ] Loading, error, empty, and partial-data states are clear.
- [ ] The dashboard is usable by keyboard and on supported responsive layouts.
