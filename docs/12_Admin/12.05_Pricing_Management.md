# 12.05 Pricing Management

**Version:** 1.0  
**Module:** Admin  
**Status:** Planned

## 1. Purpose

Provide controlled visibility and presentation management for the real InstantViral package catalog. This module must preserve catalog truth: package IDs, names, quantities, USD prices, and popular flags originate in `data/pricing/instantviral-products.ts`.

## 2. Source of Truth and Guardrails

The current catalog is mapped into `PricingPackage` records by `data/pricing/packages.ts`: prices convert from major USD units to minor units, and package-to-service matching uses platform and category.

- [ ] Never fabricate, rename, reprice, or create package tiers in the admin interface.
- [ ] Treat catalog synchronization/change as a reviewed code or controlled import operation, not a normal editor field.
- [ ] Only expose packages whose service mapping, `active` flag, and `availability === 'active'` permit display.
- [ ] Preserve the catalog's `popular` flag; presentation may not mark a different package “most popular” without a source change.

## 3. Pricing Table

Route: `/admin/pricing`. The table is read-first and lists real packages with filters for platform, service/category, active state, availability, popular flag, quantity, and price range.

| Column | Content |
| --- | --- |
| Package | Catalog ID and exact InstantViral package name. |
| Service | Resolved service name and slug. |
| Quantity | Catalog count and customer-facing label. |
| Price | USD price formatted from stored minor units. |
| Badge | `most-popular` only when catalog `popular` is true. |
| Availability | Active/inactive and availability state. |
| Updated/source | Catalog source/version and last controlled sync. |
| Actions | View contextual editor and preview. |

## 4. Package Editor

The package editor is scoped to a catalog package. Read-only source fields are package ID, service mapping, name, quantity, currency, base price, and popular flag. It may configure permitted presentation and availability metadata only when it does not alter the underlying commercial catalog.

### 4.1 Overview tab

Shows source fields, resolved service, availability, display order, current badge, and catalog provenance. Explain why unavailable packages are excluded from customer views.

### 4.2 Display tab

Manage allowed presentation fields such as a customer-safe quantity label, delivery-time copy, feature copy, and approved display order. All content is validated and must not claim unavailable guarantees.

### 4.3 Availability tab

Manage `active` and availability state with an effective date/reason if operational policy requires it. Deactivation affects new checkout selection only; it must not rewrite historical orders or receipts.

### 4.4 Service placement tab

Shows the package's eligible service and whether it appears in that service's pricing selection. A package cannot be attached to a service with a different platform/category.

### 4.5 Audit tab

Shows source sync events and permitted administrative visibility/presentation changes. Include actor, timestamp, before/after values, and reason.

## 5. PricingCard Preview

The preview renders the actual `PricingCard` used by commerce with a `PricingCardModel` built from the resolved package. It displays exact title, quantity, currency-formatted price, badge, delivery/feature copy, availability, and selected state.

- [ ] Render desktop and narrow layouts.
- [ ] Use the production formatter and current currency registry; USD is the catalog source/default.
- [ ] Make unavailable state unmistakable and disable purchase selection in preview.
- [ ] Compare preview package identity against the resolver before rendering.

## 6. Acceptance Checklist

- [ ] Every listed package can be traced to an InstantViral catalog ID.
- [ ] Staff cannot fabricate or directly edit catalog quantity, price, name, or popularity.
- [ ] Package visibility follows active/availability and compatible service mapping rules.
- [ ] PricingCard preview matches the customer-facing component and formatting.
- [ ] All permitted edits and catalog updates are auditable.
