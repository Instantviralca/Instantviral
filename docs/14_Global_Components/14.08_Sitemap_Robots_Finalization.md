# InstantViral v2

# Document 14.08 — Sitemap & Robots Finalization

Version: 1.0
Status: Production Specification

## Purpose

Finalize the XML sitemap and robots rules for InstantViral before launch.

The system must:

- Include only approved, active, indexable public pages
- Exclude skipped services
- Exclude private, transactional, admin, preview, and error routes
- Keep sitemap URLs aligned with canonical URLs
- Keep robots.txt aligned with metadata-level robots directives
- Support future Learn content without requiring a full rewrite

---

## Production Domain

Use:

https://instantviral.ca

All sitemap URLs must be absolute HTTPS URLs.

Do not use localhost, preview domains, Vercel preview URLs, or alternate hostnames.

## Track Order Rules

The public form page:

- /track-order

may be indexable.

Private order lookup responses must:

- Never create crawlable public URLs
- Never appear in sitemap
- Never expose order IDs or email addresses in metadata
- Use noindex where a private result surface exists
- Remain protected by verification and rate limiting

---

## Sitemap Entry Model

Each sitemap entry should support:

- url
- lastModified
- changeFrequency (optional)
- priority (optional)

Use conservative values.

Do not attempt to manipulate rankings through artificial priority values.

---

## Recommended Priority Guidance

Homepage:

- 1.0

Active service pages:

- 0.8 to 0.9

FAQ, About, Contact:

- 0.6 to 0.7

Legal pages:

- 0.3 to 0.5

Track Order:

- 0.4 to 0.6

These are organizational hints only and should not be treated as ranking controls.

---

## Change Frequency Guidance

Use only if helpful.

Suggested:

- Homepage: weekly
- Service pages: weekly or monthly
- FAQ: monthly
- About and Contact: monthly
- Legal pages: yearly or when changed
- Track Order form: monthly

Do not use unrealistic values such as always or hourly for static pages.

---

## Last Modified Rules

Use meaningful `lastModified` values from:

- Content update timestamps
- Service registry update timestamps
- Legal policy revision dates
- Article update dates in the future

Do not set every page to the current time during every build unless the content genuinely changed.

Avoid fake freshness signals.

## Next.js Sitemap Architecture

Use:

- `app/sitemap.ts`

The sitemap should be generated from shared registries:

- Active route registry
- Service registry
- Company/legal page registry
- Future Learn registry

Do not manually duplicate the route list in multiple files.

---

## Robots.txt Architecture

Use:

- `app/robots.ts`

Recommended base behavior:

- Allow public indexable pages
- Disallow transactional, private, admin, API, preview, and search routes
- Reference the production sitemap URL
- Use the production host where appropriate

---

## Recommended Disallow Rules

Disallow:

- /cart
- /checkout
- /order-success
- /admin/
- /api/
- /preview/
- /draft/
- /search
- Private order result routes, if any

Do not disallow CSS, JavaScript, images, fonts, or assets required for rendering public pages.

---

## Robots.txt and Noindex Coordination

Important:

- robots.txt controls crawling
- page metadata controls indexation
- authentication controls access

Do not rely on robots.txt to protect private data.

Sensitive or private pages require:

- Authentication or verification
- Noindex metadata
- No public sitemap entry
- No internal crawlable links where inappropriate

---

## Canonical Coordination

Every sitemap URL must exactly match the page canonical URL.

Validate:

- Protocol
- Host
- Path
- Trailing slash policy
- Letter casing
- Route status

If a sitemap URL and canonical URL differ, report it as an error.

---

## Internal Linking Coordination

Every indexable sitemap page should have at least one internal link from another crawlable page.

Detect and report:

- Sitemap pages with no internal links
- Internal links to noindex pages
- Internal links to skipped service routes
- Canonical pages absent from sitemap
- Sitemap pages absent from the route registry
