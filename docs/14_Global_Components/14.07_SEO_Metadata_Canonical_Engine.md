# InstantViral v2

# Document 14.07 — SEO Metadata & Canonical Engine

Version: 1.0
Status: Production Specification

## Purpose

Create one centralized SEO metadata engine for the entire InstantViral website.

The engine should manage:

- Page titles
- Meta descriptions
- Canonical URLs
- Open Graph metadata
- Twitter/X cards
- Robots directives
- Metadata fallbacks
- Duplicate metadata detection
- Missing metadata reporting
- Page-level SEO validation

The system should use the Next.js App Router Metadata API and the existing InstantViral SEO architecture.

---

## Objectives

The engine should:

- Keep metadata data-driven
- Generate unique metadata for every indexable page
- Produce self-referencing canonicals
- Prevent invalid or duplicate canonical URLs
- Support static and dynamic routes
- Keep social metadata consistent
- Prevent indexation of transactional and private routes
- Detect missing or duplicated titles and descriptions
- Keep metadata aligned with visible page content
- Support future Learn articles and platform hubs

## Title Rules

Each indexable page must have a unique, accurate title.

Recommended structure:

- Homepage: brand-focused title
- Service page: primary commercial query + brand
- Company page: page purpose + brand
- Legal page: policy name + brand
- Learn article: article title + brand

Examples:

- `Buy Instagram Followers Canada | InstantViral`
- `Buy TikTok Views Canada | InstantViral`
- `Contact InstantViral Canada | Customer Support`
- `Privacy Policy | InstantViral Canada`

Rules:

- Match the visible page purpose
- Keep titles concise
- Avoid repeating keywords
- Avoid boilerplate-heavy titles
- Do not use misleading urgency or guarantees
- Do not create identical titles for different routes

---

## Meta Description Rules

Each important indexable page should have a unique description.

Descriptions should:

- Accurately summarize the page
- Match visible content
- Use natural language
- Avoid fake claims
- Avoid keyword stuffing
- Avoid duplicated templates when specific copy exists
- Keep service refund/refill wording conditional
- Avoid listing unavailable payment methods

Google may generate a different snippet from page content, so descriptions should be treated as useful summaries rather than guaranteed displayed text.

---

## Canonical URL Rules

Every indexable canonical page should use an absolute self-referencing canonical URL.

Examples:

- `https://instantviral.ca/`
- `https://instantviral.ca/buy-instagram-followers`
- `https://instantviral.ca/privacy-policy`

Canonical rules:

- Use HTTPS
- Use the production hostname
- Use approved clean routes
- Avoid query parameters
- Avoid fragments
- Avoid trailing-slash inconsistencies
- Do not canonicalize unrelated pages to the homepage
- Do not point active service pages to platform pages
- Do not include inactive or skipped routes
- Keep sitemap and canonical URLs consistent

---

## Query Parameters

For tracking parameters such as UTM values:

- Keep the canonical URL on the clean route
- Do not generate unique canonical URLs for campaign parameters

For future search, filter, sort, or pagination parameters:

- Define explicit indexing and canonical rules before launch
- Do not assume every parameterized URL should be indexed

## Robots Directives

### Indexable

Typically:

- Homepage
- Active service pages
- About
- Contact
- FAQ
- Legal pages
- Future Learn pages

### Noindex

Use `noindex` for:

- Cart
- Checkout
- Order success
- Private order details
- Admin routes
- Search-result pages when appropriate
- Development or preview routes
- Error pages
- Duplicate or incomplete pages

The public Track Order form page may be indexable if desired, but individual lookup results and private order data must never be indexable.

---

## Transactional and Private Routes

Recommended directives:

- `/cart`: noindex, follow
- `/checkout`: noindex, nofollow or follow according to final SEO policy
- `/order-success`: noindex, nofollow
- `/admin/*`: noindex, nofollow and authentication protected
- Private order lookup responses: noindex and never exposed as crawlable static URLs

Robots metadata does not replace authentication or authorization.

---

## Default Fallbacks

Create safe fallbacks for:

- Missing title
- Missing description
- Missing OG image
- Missing Twitter image
- Missing locale

However:

- Fallbacks should not hide missing production metadata
- The validation report must still flag missing page-specific values
- Service pages must not ship with generic homepage metadata

## Duplicate Detection

Detect:

- Duplicate titles
- Duplicate descriptions
- Duplicate canonical URLs
- Multiple active entries for the same route
- Canonical URLs assigned to different pages
- Repeated generic metadata on service pages
- Missing titles
- Missing descriptions
- Missing canonical values
- Missing OG images
- Non-indexable pages appearing in indexable registries

Report issues with route and source-file details.

## Page Content Alignment

Metadata must align with visible content.

Examples:

- The service title must match the actual service
- The description must not mention unavailable features
- The title must not use a different country or service
- Refund and refill claims must remain conditional
- YouTube metadata must not imply monetization guarantees
- Page canonicals must match mounted production routes

---

## Structured Data Coordination

Metadata and schema should share:

- Canonical page URL
- Page title
- Page description where appropriate
- Organization identity
- Primary image
- Breadcrumb URLs

Do not maintain conflicting domain or URL values across metadata and schema systems.

## Internationalization Readiness

Prepare for future support of:

- Locale-specific metadata
- `alternates.languages`
- Hreflang
- Localized canonicals
- Localized OG locale values

Do not output hreflang until genuine alternate-language pages exist.

## Error and Not-Found Metadata

- Use a clear title
- Use noindex
- Avoid canonicalizing to unrelated content

Error and preview routes should not inherit misleading service metadata.

## Performance

- Generate metadata server-side
- Avoid client-side head mutation
- Reuse cached registry data where safe
- Avoid repeated metadata fetches
- Keep social images optimized
- Prevent metadata generation from blocking unnecessarily

## Security

- Never expose private order data in metadata
- Never include customer names, emails, usernames, or URLs in public metadata
- Never expose admin details
- Sanitize dynamic titles and descriptions
- Validate canonical paths against the approved route registry
