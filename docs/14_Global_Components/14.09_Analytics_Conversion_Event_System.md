# InstantViral v2

# Document 14.09 — Analytics & Conversion Event System

Version: 1.0
Status: Production Specification

## Purpose

Create one centralized analytics and conversion event system for the InstantViral website.

The system should track meaningful customer and admin actions without tightly coupling the application to a single analytics provider.

It should support:

- Google Analytics 4
- Google Tag Manager
- Microsoft Clarity
- Future analytics providers
- Internal event logging
- Consent-aware tracking

No analytics provider should be enabled unless it is actually configured.

---

## Objectives

The analytics system should:

- Standardize event names
- Standardize event payloads
- Prevent duplicate event firing
- Support consent-aware tracking
- Keep personally identifiable information out of analytics events
- Track the complete customer journey
- Support service and package performance reporting
- Support future A/B testing
- Support admin analytics
- Remain provider-independent

---

## Analytics Architecture

Recommended flow:

User Action

↓

Typed Analytics Event

↓

Analytics Dispatcher

↓

Consent and Privacy Check

↓

Enabled Provider Adapter

↓

Provider Event

The application should send one internal event.

Provider adapters translate that event into the format required by each enabled analytics platform.

## Event Data Model

Each event should support:

- eventName
- category
- action
- label (optional)
- value (optional)
- currency (optional)
- platform (optional)
- serviceSlug (optional)
- packageId (optional)
- pagePath
- pageType
- sessionId (internal anonymous identifier)
- timestamp
- metadata
- consentCategory

Do not include private customer information.

---

## Forbidden Analytics Data

Never send:

- Customer email
- Full name
- Phone number
- Social media username
- Profile URL
- Post URL
- Video URL
- Channel URL
- Order configuration text
- Custom comments
- Payment card details
- Full transaction identifiers
- Internal admin notes
- Passwords
- Authentication tokens

Order IDs should not be sent unless safely anonymized or transformed according to the approved privacy policy.

---

## Consent Categories

Support:

- Essential
- Analytics
- Marketing

Essential operational logging may function without optional analytics consent where legally appropriate.

Analytics events should only be sent to optional analytics providers when the required consent state allows it.

Marketing events should only be sent when marketing consent is valid.

## Platform-Specific Events

Existing specific event names may be mapped to the shared event system.

Examples:

- `ig_followers_package_select`
- `tiktok_views_checkout_start`
- `facebook_page_likes_url_submit`
- `youtube_subscribers_view_packages`

Recommended approach:

Use a standard event name plus structured fields.

Example:

Event:
`package_select`

Payload:

- platform: instagram
- serviceSlug: buy-instagram-followers
- packageId: real package ID

Keep legacy names only when required for an existing analytics implementation.

## Purchase Event

Track successful purchases only after server-side payment confirmation or verified successful order creation.

Recommended event:

`purchase`

Payload may include:

- anonymous transaction reference
- currency
- total
- coupon code identifier only when privacy-approved
- item list
- platform
- serviceSlug
- packageId
- quantity

Do not fire purchase events based only on a button click or unverified client redirect.

Prevent duplicate purchase events.

---

## Event Naming Convention

Use:

- lowercase
- snake_case
- clear action names
- stable event names
- consistent categories

Avoid:

- Spaces
- Random abbreviations
- Page-specific duplicates of the same action
- Event names containing user data

---

## Event Validation

Validate:

- Event name is approved
- Required fields exist
- Currency uses a supported code
- Numeric values are valid
- Platform is supported
- Service slug exists
- Package ID exists where required
- No forbidden fields are present
- Consent allows dispatch
- Event is not duplicated

Invalid events should be blocked in production and reported in development.

---

## Duplicate Event Prevention

Prevent duplicate events caused by:

- React Strict Mode
- Re-renders
- Route transitions
- Button double-clicks
- Payment return-page refresh
- Form retries
- Multiple analytics providers

Use:

- Event IDs
- Idempotency keys
- Session-level guards
- Purchase confirmation state
- Server-side verification for conversions

---

## Provider Adapters

Prepare adapters for:

- Google Analytics 4
- Google Tag Manager
- Microsoft Clarity
- Internal console/debug adapter
- Future providers

Each adapter should implement a shared interface.

Example methods:

- initialize()
- trackPageView()
- trackEvent()
- trackConversion()
- setConsent()
- reset()

Do not initialize an adapter without verified configuration.

## Environment Configuration

Use environment variables and configuration for:

- Analytics enabled
- Provider enabled
- GA4 measurement ID
- GTM container ID
- Clarity project ID
- Debug mode
- Consent mode
- Environment name

Do not expose private secrets.

## Development Debugging

Create a development-only analytics debug view.

Display:

- Event name
- Sanitized payload
- Consent result
- Providers selected
- Duplicate status
- Validation errors

Never expose this panel in production.

---

## Error Handling

Analytics failures must not break:

- Page rendering
- Package selection
- Cart
- Checkout
- Order tracking
- Forms

Provider errors should be isolated and logged safely.

---

## Performance

- Lazy-load optional provider scripts
- Do not block page rendering
- Avoid unnecessary event listeners
- Avoid sending duplicate page views
- Batch events only when supported and safe
- Respect Core Web Vitals
- Load marketing scripts only after consent where required

---

## Privacy

The analytics implementation must align with:

- Privacy Policy
- Cookie Policy
- Consent settings
- Data minimization
- Retention rules
- Enabled providers

Do not publish provider-specific disclosures until the provider is genuinely enabled.

Prepare safe reporting dimensions for:

- Platform
- Service
- Package
- Page type
- Traffic source
- Device category
- Country or region only when collected lawfully
- Coupon usage
- Payment method category
- Order status category

## Conversion Funnel

Recommended funnel:

1. Homepage or landing page view
2. Service page view
3. Package impression
4. Package selection
5. Order configuration submit
6. Add to cart
7. Cart view
8. Checkout start
9. Checkout submit
10. Verified purchase

Track abandonment between stages without identifying individual customers.
