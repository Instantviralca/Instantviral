# InstantViral v2

## Pricing System

Version: 1.0

Status: Production Architecture

Purpose: Define a reusable, API-driven pricing architecture that supports every InstantViral platform and service without hardcoded pricing.

---

# Objectives

The pricing system must support:

- Multiple package sizes per service
- Future discounts and coupon rules
- Multiple currencies
- Subscription-ready pricing
- API-driven data delivery
- Reuse across platforms, services and future products

Pricing values must never be hardcoded in page components.

---

# Package Structure

Every purchasable package must provide:

- Package ID
- Service ID
- Title
- Quantity
- Price
- Compare-at price when applicable
- Currency
- Delivery estimate
- Features
- Badge
- Stock state
- Display order
- Active status

Packages must be ordered by quantity in ascending order.

---

# Example Data Model

```ts
interface PricingPackage {
  id: string
  serviceId: string
  title: string
  quantity: number
  price: number
  compareAtPrice?: number
  currency: 'USD' | 'EUR' | 'GBP' | 'PKR' | 'CAD' | 'AUD'
  deliveryEstimate: string
  features: string[]
  badge?: 'Most Popular' | 'Best Value' | 'Recommended' | 'Starter' | 'Premium'
  stockState: 'Active' | 'Hidden' | 'Out of Stock'
  displayOrder: number
  subscriptionEligible?: boolean
}
```

---

# Package Badges

Supported badges:

- Most Popular
- Best Value
- Recommended
- Starter
- Premium

A badge is optional and must be driven by package data. It should communicate value without obscuring the package title, quantity or price.

---

# Delivery Times

Each package must expose a clear delivery estimate. Delivery text is service-specific and must be supplied by pricing data, for example:

- Starts within minutes
- Delivered within 1–3 days
- Delivered gradually over 7 days

Delivery estimates must be visible before a customer begins checkout.

---

# Features

Package features describe what is included with the selected quantity. Feature lists must be data-driven, concise and scannable. They may include service quality, refill policy, delivery method or support details where applicable.

---

# Card Layout

Each pricing card should contain:

- Optional badge
- Package title
- Quantity
- Current price
- Optional compare-at price and savings context
- Delivery estimate
- Feature list
- Order Now CTA

The primary CTA label is `Order Now`.

---

# Currencies

USD is the default currency.

Supported currencies:

- USD
- EUR
- GBP
- PKR
- CAD
- AUD

Currency formatting, conversion and selection must be centralized so a package can be presented consistently across the application.

---

# Discount Rules

The architecture must allow future discount rules without changing package UI components. Rules may include:

- Package-level discounts
- Volume discounts
- Coupon discounts
- Currency-specific promotions
- Subscription discounts
- Time-limited campaigns

A discount must preserve the original package price and return a calculated final price for display and checkout.

---

# Stock States

Supported stock states:

- Active — available for purchase
- Hidden — excluded from customer-facing pricing displays
- Out of Stock — visible when useful but unavailable for purchase

Out-of-stock packages must not allow an order to be added to the cart.

---

# Responsive Behavior

Pricing grids must use:

- 3–4 cards per row on large screens
- 2 cards per row on tablet screens
- 1 card per row on mobile screens

Cards must maintain a readable hierarchy and full-width CTA on small screens.

---

# Accessibility

Pricing experiences must provide:

- Semantic headings and list markup
- Keyboard-accessible CTAs
- Visible focus states
- Sufficient color contrast
- Text alternatives for non-text price or status indicators
- Clear out-of-stock messaging

---

# Analytics

Track pricing interactions including:

- Pricing section viewed
- Package viewed
- Package selected
- Order Now clicked
- Currency changed
- Discount applied
- Out-of-stock package encountered

Analytics events must include package ID, service ID and selected currency where available.

---

# Data Source

Pricing data must be loaded from `data/pricing/` during the static implementation or from an API when backend pricing is available. UI components consume normalized package data and must not embed price values.

---

# Future Integrations

The architecture must accommodate:

- Pricing APIs
- Coupon and promotion engines
- Currency conversion services
- Subscription billing
- Inventory or supplier availability
- A/B testing
- Customer-specific pricing

---

# Acceptance Checklist

- [ ] Pricing is data-driven and never hardcoded in UI components
- [ ] Packages support all required fields and badges
- [ ] Quantities are displayed in ascending order
- [ ] USD is the default currency and all listed currencies are supported
- [ ] Discount rules can be added without rebuilding card components
- [ ] Active, Hidden and Out of Stock states are enforced
- [ ] Order Now is the primary package CTA
- [ ] Grid responds at 3–4, 2 and 1 columns
- [ ] Accessibility and analytics requirements are implemented
- [ ] Data can come from `data/pricing/` or an API
