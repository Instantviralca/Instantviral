# InstantViral v2

## Pricing Packages UI

Version: 1.0

Status: Defined

Purpose: Define the reusable pricing packages interface displayed below the service-page hero.

---

# Placement

The pricing packages section appears below the hero. It introduces purchasable package options with clear value, delivery expectations and a direct path to order configuration.

---

# Layout

The section must use a responsive pricing grid:

- 3 cards per row on large screens when space allows
- 2 cards per row on tablet screens
- 1 card per row on mobile screens

Packages are rendered in ascending quantity order. The selected or recommended package may receive visual emphasis without reducing readability of other options.

---

# Pricing Card Structure

Every card includes:

- Optional package badge
- Package title
- Quantity
- Current price
- Optional compare-at price
- Delivery estimate
- Package features
- Order Now CTA

The compare-at price must be visually secondary and clearly distinguishable from the current price. The CTA forwards the selected package to order configuration.

---

# Components

The UI is composed from the following reusable components:

- `PricingPackagesSection` — section heading, context and data boundary
- `PricingGrid` — responsive package grid
- `PricingCard` — single package presentation
- `PackageBadge` — optional value or recommendation label
- `PriceDisplay` — current and compare-at pricing
- `DeliveryEstimate` — delivery timing text
- `PackageFeatures` — included-feature list
- `PricingCTA` — Order Now action

---

# States

## Loading

Use pricing card skeletons while package data is loading. Skeletons should preserve the expected card dimensions and avoid layout shift.

## Empty

When no packages are available, show a concise empty state that explains availability and provides a clear next action, such as returning to services or contacting support.

## Out of Stock

Out-of-stock packages may be displayed when supplied by pricing data, but their CTA must be disabled or replaced with an unavailable message.

---

# Content Rules

- Use package data as the sole source of pricing content.
- Do not hardcode amounts, features or delivery estimates in components.
- Keep feature lists short and scannable.
- Preserve the Order Now CTA label for available packages.
- Do not make compare-at pricing appear as the current payable amount.

---

# Accessibility

Cards must use semantic structure, readable heading order, keyboard-accessible CTAs, visible focus states and sufficient contrast. Badge, price and availability meaning must not depend on color alone.

---

# Acceptance Checklist

- [ ] Section appears below the hero
- [ ] Grid supports 3, 2 and 1 column layouts
- [ ] Cards contain badge, title, quantity, price, compare-at price, delivery, features and CTA where applicable
- [ ] Components match the defined reusable component list
- [ ] Loading skeleton and empty state are available
- [ ] Out-of-stock packages cannot be ordered
- [ ] UI is data-driven, accessible and responsive
