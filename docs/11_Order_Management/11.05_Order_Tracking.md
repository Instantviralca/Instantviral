# 11.05 Order Tracking

**Version:** 1.0  
**Module:** Order Management  
**Status:** Planned

## 1. Purpose

Provide a public, privacy-preserving order lookup experience at `/track-order` so customers can view the current progress of an order without creating an account.

## 2. Public Route

**Route:** `/track-order`

The route presents a verification form and, only after successful verification, a concise order status view. Links in transactional notifications may prefill the order ID, but email verification remains required before protected order details are shown.

## 3. Verification Flow

1. Customer enters an Order ID and the email address used at checkout.
2. The server normalizes the values and performs a constant-shape verification request.
3. The system verifies that the email matches the order record.
4. On success, the system creates a short-lived, scoped tracking session or signed token.
5. The customer is redirected to the verified tracking view.
6. The tracking session expires and must be re-verified before further access.

The response for an unknown Order ID and an incorrect email must be indistinguishable to prevent order enumeration.

## 5. Customer-visible Statuses

| Status | Customer label | Description |
| --- | --- | --- |
| `pending_payment` | Awaiting payment | Payment has not yet been confirmed. |
| `paid` | Payment confirmed | The order is ready for processing. |
| `processing` | In progress | Delivery work is underway. |
| `partially_completed` | Partially delivered | Some of the order is complete; remaining fulfilment is in progress. |
| `completed` | Completed | The order has been delivered. |
| `cancelled` | Cancelled | The order will not be fulfilled. |
| `refunded` | Refunded | A refund has been completed or recorded. |
| `action_required` | Action required | The customer must contact support or provide requested information. |

Internal provider statuses and error messages must be mapped to safe customer-facing language.

## 6. Timeline

The timeline is derived from immutable order events and displays only relevant, customer-safe events in chronological order:

1. Order placed
2. Payment confirmed
3. Processing started
4. Partial delivery (when applicable)
5. Delivery completed, cancellation, or refund outcome

Each item contains a label, plain-language description, and timestamp. The active step is visually distinct. Events that reveal internal operations, staff identities, provider responses, or risk signals are never shown publicly.

## 7. Security Controls

- [ ] Require both Order ID and checkout email before returning order data.
- [ ] Use generic failed-verification responses.
- [ ] Rate limit by IP, normalized email, and order identifier.
- [ ] Add progressive throttling and abuse monitoring for repeated failures.
- [ ] Use HTTPS and server-side validation for all input.
- [ ] Issue short-lived, httpOnly, secure tracking sessions or signed scoped tokens.
- [ ] Prevent tracking pages from being indexed; set `noindex, nofollow`.
- [ ] Avoid placing customer email or sensitive order data in URLs, analytics, or logs.
- [ ] Return only the minimum fields required for tracking.

## 8. Accessibility and UX

- [ ] Associate labels, validation messages, and inputs programmatically.
- [ ] Make timeline progress understandable without color alone.
- [ ] Support keyboard-only completion and screen-reader announcements for errors.
- [ ] Provide concise, helpful copy for pending and delayed states.
- [ ] Keep mobile layouts readable and actions easy to reach.

## 9. Integration Points

- **Order service:** verified order lookup and customer-safe status projection.
- **Payment service:** payment confirmation and refund state.
- **Fulfilment service:** processing and delivery milestones.
- **Notification system:** links to `/track-order` and shared status wording.
- **Support tooling:** secure handoff for unresolved tracking inquiries.
- **Analytics:** aggregated, privacy-safe funnel and verification metrics.

## 10. Acceptance Checklist

- [ ] A valid Order ID plus matching email displays the correct order projection.
- [ ] Invalid ID and invalid email receive the same public failure response.
- [ ] Unverified requests cannot access tracking details.
- [ ] Expired sessions require fresh verification.
- [ ] Timeline events appear in correct order and use customer-safe language.
- [ ] The route is responsive, accessible, and excluded from search indexing.
