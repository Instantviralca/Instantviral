# 11.04 Notification System

**Version:** 1.0  
**Module:** Order Management  
**Status:** Planned

## 1. Purpose

Define the email-first notification system that keeps customers informed throughout an order lifecycle while providing an auditable delivery record for support and operations.

## 2. Goals

- [x] Make transactional email the default customer communication channel.
- [x] Send clear, timely updates for meaningful order events.
- [x] Preserve a complete notification history per order.
- [x] Surface delivery failures for support follow-up.
- [x] Allow future channels without changing order-event producers.

## 3. Channel Strategy

### 3.1 Email-first delivery

Email is the required launch channel. Messages are sent to the email captured with the order and use branded, responsive templates.

### 3.2 Future channels

The notification service must support channel adapters for future use:

- In-app notifications
- SMS
- WhatsApp or other messaging platforms
- Webhooks for partner or API customers

Channel preference, consent, and opt-in requirements must be evaluated before enabling non-email channels.

## 4. Notification Triggers

| Order event | Customer notification | Notes |
| --- | --- | --- |
| Order created | Order confirmation | Includes order ID, purchase summary, and tracking link. |
| Payment received | Payment confirmation | Sent only after successful payment capture. |
| Payment failed | Payment issue | Provides a safe path to retry or contact support. |
| Order queued | Processing started | Optional operational reassurance for longer fulfilment windows. |
| Order processing | Order in progress | Explains that delivery has begun. |
| Order completed | Delivery complete | Includes completion summary and support link. |
| Order partially completed | Partial delivery update | States delivered and remaining quantities where applicable. |
| Order cancelled | Cancellation confirmation | Includes refund guidance when relevant. |
| Refund initiated | Refund initiated | States expected processing window without guarantees. |
| Refund completed | Refund confirmation | Includes amount and original payment method reference when available. |
| Support action required | Action required | Used only for verified customer input requirements. |

## 5. Templates

### 5.1 Template requirements

Every template must include:

- InstantViral branding and a clear transactional subject line.
- Customer-safe order reference; never expose internal IDs or provider credentials.
- A primary call to action when the customer can take an action.
- A `/track-order` link prefilled with the order ID where appropriate.
- Support contact details and required legal/footer content.
- Plain-text and responsive HTML variants.

### 5.2 Template catalog

- [ ] `order-confirmation`
- [ ] `payment-confirmation`
- [ ] `payment-failed`
- [ ] `order-processing`
- [ ] `order-completed`
- [ ] `order-partially-completed`
- [ ] `order-cancelled`
- [ ] `refund-initiated`
- [ ] `refund-completed`
- [ ] `action-required`

Templates receive a typed, minimal payload: customer name (when available), order ID, product summary, quantities, status, relevant timestamps, and safe links.

## 6. Delivery Workflow

1. An order or payment service emits a domain event after its transaction commits.
2. The notification service validates the event and resolves eligible recipients and channels.
3. A notification record is created with `queued` status before delivery is attempted.
4. The channel adapter renders the current approved template and submits the message to the provider.
5. Provider callbacks update the record with delivery events and failure details.
6. Retriable failures follow controlled retry rules; non-retriable failures are visible to staff.

## 7. Delivery Statuses

| Status | Meaning |
| --- | --- |
| `queued` | Created and awaiting a delivery attempt. |
| `sending` | Submitted to the channel adapter. |
| `sent` | Accepted by the provider. |
| `delivered` | Confirmed delivered when the provider supports it. |
| `opened` | Open event received; informational only. |
| `failed` | A final delivery failure occurred. |
| `suppressed` | Skipped due to an invalid, bounced, unsubscribed, or blocked address. |
| `cancelled` | Intentionally stopped before delivery. |

A `sent` status must not be represented as `delivered` unless a provider delivery event confirms it.

## 8. Retry and Failure Handling

- [ ] Retry transient provider/network errors with bounded exponential backoff.
- [ ] Prevent duplicate customer messages with an idempotency key based on order, trigger, and event version.
- [ ] Do not retry permanent recipient failures such as invalid addresses or hard bounces.
- [ ] Record the provider message ID, response code, failure category, and timestamps.
- [ ] Alert operations when delivery failure rates exceed configured thresholds.
- [ ] Allow authorized staff to resend eligible transactional notifications from the order record.

## 9. Notification History

Each order exposes an internal notification history containing:

- Notification type and template version
- Channel and masked recipient
- Triggering event and idempotency key
- Queued, sent, delivered, and failed timestamps
- Current delivery status and provider message ID
- Retry count and failure reason
- Actor for manual sends or cancellations

History is append-oriented. Staff annotations may be added without overwriting provider events.

## 10. Integration Points

- **Order service:** lifecycle events, order metadata, and tracking references.
- **Payment service:** successful payment, failures, refunds, and chargeback-related signals.
- **Fulfilment providers:** processing, partial completion, completion, and failure events.
- **Email provider:** message submission, bounce, delivery, and complaint webhooks.
- **Admin module:** notification history, resend controls, template preview, and operational filters.
- **Support tooling:** secure order lookup and customer communication context.
- **Observability:** structured logs, delivery metrics, provider latency, and alerting.

## 11. Security and Privacy

- [ ] Authenticate and verify inbound provider webhooks.
- [ ] Store only the notification data needed for delivery and audit.
- [ ] Mask email addresses in standard staff views.
- [ ] Keep tokens and provider credentials in server-side secret storage.
- [ ] Do not include passwords, payment details, or fulfilment-provider credentials in notifications.
- [ ] Use signed, expiring links where a notification grants access to protected information.
- [ ] Apply retention and deletion policies to notification content and events.

## 12. Acceptance Checklist

- [ ] All defined order events create at most one idempotent notification per recipient and channel.
- [ ] Email templates render correctly in HTML and plain text.
- [ ] Delivery callbacks update notification status without changing order state unexpectedly.
- [ ] Failed deliveries are searchable and actionable in admin.
- [ ] Order history provides a complete, timestamped audit trail.
- [ ] Notification failures do not block a successfully committed order transaction.
