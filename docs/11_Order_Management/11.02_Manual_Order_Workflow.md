# InstantViral v2

# Document 11.02 — Manual Order Workflow

Version: 1.0

Status: Production Specification

---

# Purpose

This document defines the complete manual order fulfillment workflow for InstantViral.

Unlike automated SMM panels, Version 1 of InstantViral will use a manual fulfillment process. Every order will be reviewed and completed manually through the Admin Panel.

This workflow should remain compatible with future automation if SMM provider integrations are added later.

---

# Objectives

The workflow should:

- Keep order processing simple.
- Prevent missed orders.
- Provide clear order statuses.
- Keep customers informed.
- Allow future automation without changing the customer experience.

---

# Order Flow

```
Customer Visits Service Page
↓
Select Package
↓
Configure Order
↓
Add to Cart
↓
Checkout
↓
Payment Successful
↓
Order Created
↓
Admin Receives New Order
↓
Manual Review
↓
Order Processing
↓
Order Completed
```

---

# Manual Processing Flow

After payment:

1. A new order is created.
2. The order appears in the Admin Panel.
3. Admin reviews:
   - Service
   - Package
   - Username or URL
   - Customer Email
   - Payment Status
4. Admin manually completes the order using the chosen fulfillment method.
5. Admin updates the order status.
6. Customer is notified.
7. Customer can track the updated status.

---

# Required Order Information

Each order should contain:

- Order ID
- Customer Email
- Platform
- Service
- Package
- Quantity
- Username or URL
- Notes (optional)
- Payment Status
- Order Status
- Created At
- Updated At

---

# Admin Actions

The administrator can:

- View Order
- Start Processing
- Mark Completed
- Mark Partial
- Cancel Order
- Refund Order (future)
- Add Internal Notes

Internal notes must never be visible to customers.

---

# Order Status Flow

```
Pending
↓
Processing
↓
Completed
```

Alternative paths:

- Pending → Cancelled
- Processing → Partial
- Processing → Cancelled
- Completed → Refunded (future)

---

# Notifications

The system should trigger notifications when:

- Order Created
- Processing Started
- Order Completed
- Order Cancelled
- Order Refunded (future)

Notification delivery is handled by the Notification System.

---

# Order History

Every status update should be recorded.

Each timeline entry should include:

- Timestamp
- Status
- Updated By
- Optional Note

Timeline history should never be deleted.

---

# Error Handling

Examples:

- Invalid username
- Duplicate order
- Customer requests correction
- Payment verification issue

The workflow should allow administrators to resolve these cases without deleting the order.

---

# Customer Experience

Customers should:

- Receive an order confirmation.
- Be able to track their order.
- See the current order status.
- Receive a completion notification.

Customers cannot directly modify an order after payment.

---

# Future Compatibility

The manual workflow should allow future migration to automated fulfillment.

The transition should only replace the fulfillment step while preserving:

- Order IDs
- Statuses
- Notifications
- Tracking
- Customer experience

---

# Analytics

Track:

- New orders
- Orders completed
- Orders cancelled
- Average completion time
- Average processing time

---

# Security

- Only authorized administrators can update orders.
- Log every status change.
- Protect customer information.
- Prevent duplicate processing.

---

# Integration Points

Works with:

- Checkout
- Payment Gateway System
- Order Status System
- Notification System
- Order Tracking
- Admin Panel
