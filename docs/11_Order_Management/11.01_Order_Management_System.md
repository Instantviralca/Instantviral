# InstantViral v2

## Order Management System

Version: 1.0

Status: Architecture Defined

Purpose: Define the Order Management System (OMS) architecture for tracking an order from creation through delivery, failure, cancellation or refund.

---

# OMS Architecture

The OMS is the source of truth for order lifecycle state. It receives order, payment and fulfilment updates, records a chronological event timeline, coordinates service providers and triggers customer notifications.

The architecture must keep order state independent from any single payment gateway or fulfilment provider.

---

# Order Statuses

Supported order statuses:

- Draft
- Pending Payment
- Payment Processing
- Paid
- Queued
- Processing
- Partially Delivered
- Delivered
- Failed
- Cancelled
- Refunded

Statuses must have defined transitions and must not be changed directly without recording the actor, source and reason where relevant.

---

# Timeline Events

Each order maintains an immutable chronological timeline. Events may include:

- Order created
- Customer information updated
- Coupon applied or removed
- Payment initiated
- Payment verified
- Payment failed
- Fulfilment queued
- Provider submission created
- Delivery progress updated
- Order delivered
- Order failed or cancelled
- Refund issued
- Customer notification sent

Every event should record a timestamp, event type, source and relevant non-sensitive metadata.

---

# Provider Manager

The Provider Manager abstracts external fulfilment providers. It is responsible for:

- Selecting an eligible provider for a service
- Creating provider submissions
- Mapping provider responses to normalized OMS statuses
- Receiving and processing provider updates
- Retrying safe operations
- Recording provider reference IDs and operational events

Provider-specific details remain behind the manager boundary.

---

# Notifications

Notifications are triggered from meaningful OMS events. Initial notification categories include:

- Order confirmation
- Payment confirmation or failure
- Processing update
- Delivery completion
- Cancellation
- Refund confirmation

Notification delivery channels and templates must be configurable and must not block core order-state updates.

---

# Data and Audit Requirements

Orders must retain the selected package, customer-provided configuration, pricing snapshot, payment reference, fulfilment references and lifecycle timeline needed for support and reconciliation. Sensitive data must be minimized and protected.

---

# Current Scope

This document defines OMS architecture only. No administrative interface is included. An admin UI may be built later on top of the status, timeline, provider and notification capabilities described here.

---

- [ ] Timeline events capture chronological lifecycle activity
- [ ] Provider Manager isolates fulfilment-provider behavior
