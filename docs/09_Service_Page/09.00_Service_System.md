# InstantViral v2

# Document 09.00 — Service Page System

Version: 1.0

Status: Master Architecture

---

# Purpose

This document defines the architecture for every commercial service page on InstantViral.

Examples include:

- `/instagram/buy-instagram-followers`
- `/instagram/buy-instagram-likes`
- `/instagram/buy-instagram-views`
- `/tiktok/buy-tiktok-followers`
- `/youtube/buy-youtube-subscribers`
- `/facebook/buy-facebook-followers`

All service pages must follow this system.

---

# Objectives

Every service page should:

- Match the Design System.
- Match the Brand Strategy.
- Match the SEO System.
- Match the Copywriting System.
- Match the Homepage experience.

The visitor should always feel they are on the same website.

---

# Primary Goals

- Rank for commercial keywords.
- Answer user questions clearly.
- Build trust.
- Encourage informed purchases.
- Support AI-powered search systems with structured, well-organized content.

---

# Standard Page Structure

Every service page follows this order:

1. Hero
2. Key Benefits
3. Features
4. How It Works
5. Customer Reviews
6. FAQ
7. Related Services
8. Final CTA

This order is consistent across all service pages.

---

# Content Rules

- One primary keyword per page.
- Helpful, original copy.
- No keyword stuffing.
- Clear headings.
- Short paragraphs.
- Accurate claims only.

---

# SEO Rules

Each page must include:

- Unique title
- Unique meta description
- Canonical URL
- Open Graph data
- Twitter metadata
- Service JSON-LD
- Breadcrumb JSON-LD
- FAQ JSON-LD (when FAQs exist)

---

# Internal Linking

Every service page should link to:

- Parent platform page
- Related services
- Relevant Learn articles
- FAQ where appropriate

The platform page should also link back to all service pages.

---

# UX Principles

Users should understand:

- What the service is
- Who it is for
- Why someone would use it
- What happens next

Avoid unnecessary scrolling before presenting key information.

---

# Design Principles

- Consistent spacing
- Reusable components
- Mobile-first layouts
- Accessible interactions
- No one-off styles

---

# Accessibility

- One H1 per page
- Semantic HTML
- Keyboard navigation
- Focus-visible states
- Descriptive buttons
- Accessible forms and accordions

---

# Performance

- Optimized images
- Lazy-load non-critical media
- Minimize JavaScript
- Stable layouts
- Fast Largest Contentful Paint

---

# Analytics

Track:

- CTA clicks
- Scroll depth
- FAQ interactions
- Related service clicks
- Exit points
- Conversion funnel events

---

# Data Sources

Service pages must consume data from:

- `data/services.ts`
- `data/content/`
- `seo/`
- `schemas/`

No marketing copy should be hardcoded inside components.

---

# Component Strategy

Every section must be reusable.

A single component should support every service by receiving typed props.

Do not create page-specific component variations unless absolutely necessary.

---

# Acceptance Checklist

- Architecture is reusable.
- Content is data-driven.
- SEO system is integrated.
- Accessibility requirements are met.
- Performance targets are considered.
- Brand consistency maintained.
- Production build passes.
